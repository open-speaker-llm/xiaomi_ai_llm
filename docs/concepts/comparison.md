# 与同类开源项目对比

把本项目放到小爱音箱接入 LLM 这条赛道里横向看一遍：它和 [open-xiaoai](https://github.com/idootop/open-xiaoai)、[mi-gpt](https://github.com/idootop/mi-gpt)、[xiaogpt](https://github.com/yihong0618/xiaogpt) 各走了什么路线、强弱在哪、有哪些可以互相借鉴。

> 表中其它项目的信息基于其公开 README（采集于 2026-06），可能随上游更新而变化，以各自仓库为准。

## 一张表看清差异

| 维度 | **本项目** | open-xiaoai | mi-gpt | xiaogpt |
|---|---|---|---|---|
| **接入原理** | 串口→SSH，`/data/init.sh` 钩子，**音箱本机** shell 读小米原生 NLP 的结构化结果 | 刷机补丁固件，直接接管麦克风/扬声器 | 小米**云账号**登录，米家 API 控制 | 小米**云账号**(miservice)，轮询对话记录 |
| **需要 root/刷机** | 需要（串口拿 root，但不全量刷机，只加钩子） | **需要全量刷机** | 不需要 | 不需要 |
| **需要常驻电脑/服务器** | **不需要**（音箱自己直连 LLM+TTS） | 需要 Server 端 | 需要（PC/NAS/Docker 常驻） | 需要（PC/Docker 常驻） |
| **依赖小米云账号** | **不依赖** | 不依赖 | **依赖** | **依赖** |
| **支持设备** | 仅 MDZ-25-DA / S12A（2019 老机） | 仅 2 款新机（LX06 / OH2P） | 多数机型 | 多数机型（LX06/X10A…） |
| **路由策略** | **按 NLP domain/action 智能分流**，只接管小爱答不了的 | 全量接管 | 关键词触发 | "帮我/请问"关键词触发 |
| **LLM** | DeepSeek / Claude / MiniMax / OpenAI | 小智 / Gemini Live 等 | ChatGPT / 豆包 / Kimi 等 | 9 种 |
| **TTS** | **EdgeTTS 跑在音箱本机**（独有）/ 小爱原生兜底 | 接管扬声器 | 多种（跑在服务端） | 多种（跑在服务端） |
| **连续对话** | ✅ boot0 本地 VAD 录音（实验性）；boot1 未通（ExpectSpeech 归云端） | ✅ 真打断 | ✅ | ✅ |
| **对话记忆** | ✅ 多轮：本机历史拼接（默认 6 轮）+ server 链路 session_id | ✅ | ✅ 长短期 | ✅ |
| **维护状态** | 活跃（个人） | ⚠️ 已停更 | ⚠️ 已停更 | ✅ 活跃 |

## 三种接入哲学

这些项目本质上是**三种完全不同的接入哲学**：

1. **云账号路线**（mi-gpt / xiaogpt）：不碰硬件，用小米账号当"遥控器"——轮询云端拿到"小爱听到了啥"，再通过云端让音箱说话。**最易用**（pip 装一下 / Docker 起一下），但①依赖小米云接口，小米一改 API 或封号就废；②有云端轮询延迟；③对话过小米云。

2. **刷机接管路线**（open-xiaoai）：把音箱的耳朵嘴巴整个夺过来。**功能最强**（真打断、自定义唤醒词），但①要全量刷机有变砖风险；②只支持 2 款新机；③已停更。

3. **本机原生优先路线**（本项目）：不夺权、不轮询云，而是在**音箱本机**读小米自己的 NLP 结果，只接管它答不了的问题，LLM 和 TTS 也都在本机跑。

## 本项目真正独有的两点

- **路由最优雅**：别人要么全量接管、要么靠"帮我 / 请问"关键词猜。本项目用小米 NLP 给出的 `domain/action` **结构化字段**判断——开灯、天气、闹钟继续走原生（毫秒级、体验不打折），只有真答不上来才转 LLM。这是四者里唯一"复用小爱长板"的思路。

- **EdgeTTS 跑在音箱本机**：另外三个的 TTS 都在 PC / 服务器侧（因为它们本来就在外部跑代码）。本项目是唯一让音箱**自己**直连微软 EdgeTTS 的（见 `device/ettsc/`，纯阻塞 rustls 客户端）——这正是"音箱能跑自己代码"才有的能力。

## 定位与取舍

- **自治度最高**：四者里唯一不要电脑、不要云账号、不过小米云的方案。
- **代价是最难装**（要拆机接串口）且**只支持一款老机**。但这台老机恰恰是其它项目不太覆盖的——把自己定位成"**给被主流方案抛弃的老音箱续命**"，比硬拼设备数量更有意义。
- **维护韧性反而是隐藏优势**：open-xiaoai 和 mi-gpt 都已停更，它们依赖的小米云接口 / 固件随时可能失效却没人修。本项目不依赖云账号，**外部失效面只剩 EdgeTTS 版本号和 LLM API**，这条"少依赖"的路线长期更抗老化。

## 可借鉴 / 未来方向

1. **boot1 的连续对话仍是缺口**。boot0 已用本地 VAD 录音实现免唤醒追问，但 boot1 的免唤醒开麦设备端手段经对照实验确认不通（唯一可工作的 ExpectSpeech 归云端控制）。而 mi-gpt / xiaogpt 的连续对话恰恰是借小米云账号 API 实现的——一个混合方案值得探：平时全本机自治，**仅在 boot1 需要追问时调云账号 API 触发 ExpectSpeech**。详见 [history/followup-exploration.md](../history/followup-exploration.md)。

2. **轻量入门档**。云账号路线 `pip install` 就能用，本项目要拆机。可借云账号路线做一个低门槛入口：让没勇气拆机的人先用 miservice 方式尝鲜，确认喜欢再上"本机自治"完全体。

3. **EdgeTTS 版本过期自愈**。本机 EdgeTTS 依赖 `Sec-MS-GEC-Version`，微软抬高门槛时会 403 静默降级到原生音色。可做"403 精准报警 + 版本自动更新 + 缓存重试"，让这条少依赖路线更耐老化。
